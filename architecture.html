Step 1 — Auth
Browser fetches /speech-token and /ice-token. Flask proxies both to Azure, keeping the Speech Key server-side only.
Step 2 — WebRTC Setup
Browser creates RTCPeerConnection with Azure TURN servers, adds recv-only transceivers, then calls startAvatarAsync(peerConn).
Step 3 — Live Stream
Azure streams video + audio directly to the browser. Flask is not involved. The <video> element shows Lisa continuously once onplaying fires.
Step 4 — Chat
User speaks (browser STT) or types. Browser posts to /chat. Flask injects date + location, runs the LangChain agent, returns reply text.
Step 5 — Speech
Browser calls speakTextAsync(reply). Azure TTS synthesizes speech and animates Lisa's mouth over the existing WebRTC stream.
Step 6 — Idle
avatarEventReceived fires SwitchedToIdle when speaking ends. UI resets state. Video stream stays visible — Azure plays Lisa's idle pose.
Step-by-Step Deep Dive
How each phase works under the hood
1
Authentication & Token Fetch

Before any avatar or audio can work, the browser needs two things: a short-lived Azure Speech auth token and TURN relay credentials for WebRTC. Both requests go to Flask first — the actual Azure Speech Key is never exposed to the browser.

GET /speech-token
Flask calls Azure STS endpoint with the Speech Key in the header. Azure returns a JWT-style token valid for 10 minutes. Flask also bundles the region, avatar character, style, and voice name so the browser has everything it needs in one call.
GET /ice-token
Flask calls the Azure TTS avatar relay endpoint. Azure returns TURN server URLs, a username, and a password. These are passed as-is to the browser and used to configure RTCPeerConnection so WebRTC can punch through NAT and firewalls.
Azure endpoints called by Flask
POST https://{region}.api.cognitive.microsoft.com/sts/v1.0/issueToken
GET https://{region}.tts.speech.microsoft.com/cognitiveservices/avatar/relay/token/v1
Header: Ocp-Apim-Subscription-Key: {AZURE_SPEECH_KEY}
2
WebRTC Peer Connection Setup

With both tokens in hand, the browser builds a WebRTC peer connection. This is a direct media channel between the browser and Azure's avatar rendering infrastructure — Flask plays no role after this point for media.

Sequence inside azureInit()
a
new RTCPeerConnection({ iceServers: [{ urls, username, credential }] }) — creates the connection using Azure TURN servers from Step 1.
b
addTransceiver("video", { direction: "recvonly" }) and addTransceiver("audio", { direction: "recvonly" }) — browser declares it will only receive, never send, media tracks.
c
new SpeechSDK.AvatarSynthesizer(speechConfig, avatarConfig) — SDK is initialized with the auth token, voice, character (lisa), and style (casual-sitting). Background color is set to match the app's dark theme.
d
synthesizer.startAvatarAsync(peerConn) — SDK performs the WebRTC offer/answer exchange with Azure. Azure responds with an SDP answer and ICE candidates, completing the connection.
Why recv-only? The avatar video is rendered entirely on Azure's servers. The browser never sends any video or audio to Azure — it only receives the rendered avatar stream. This keeps bandwidth low and offloads all rendering to the cloud.
3
Live Avatar Stream

Once the WebRTC handshake completes, Azure starts delivering a continuous video and audio stream to the browser. This is a real-time rendered 3D avatar — not a pre-recorded video. Azure renders Lisa's idle pose at all times and transitions to speaking animation when speech is synthesized.

Video Track
Attached to <video id="azure-video"> via srcObject. Stays display:none until onplaying fires, then the static placeholder image is hidden and the live video takes over permanently.
Audio Track
Attached to a hidden <audio> element and also routed through the Web Audio API via createMediaStreamSource. Dual routing ensures playback works on both mobile and desktop Chrome/Edge.
Key design rule: The video is never hidden once connected. Azure streams Lisa continuously — idle when silent, speaking animation when TTS is active. Hiding the video on idle would cause a blank frame. The static images.webp is only a loading placeholder used before WebRTC connects, and as a fallback on ICE failure.
4
Chat — Voice Input or Text

The user interacts via microphone (browser's built-in SpeechRecognition API) or the text input. Either way the final message is sent to POST /chat and Flask handles all the AI logic before returning a reply.

Speech-to-Text (Browser)
Uses the Web Speech API — no Azure STT, no API key required. recognition.continuous = false and interimResults = true so partial transcripts appear in real time. On silence/no-speech the browser fires onend without a final result — the app detects this and automatically restarts the mic to keep listening.
Flask /chat Processing
Before the message reaches the LangChain agent, Flask automatically prepends two context tags:
[Today's date: 2024-01-15] {user_message}
[User's current location: Paris, France (48.8566,2.3522)] {user_message}
Location is resolved by reverse-geocoding the lat/lng the browser sends alongside the message. This makes the agent location-aware without the user needing to state their city explicitly.
LangChain Agent
The agent has access to Google Places tools (nearby search, place details) and maintains per-session memory keyed by session_id — a random ID generated once per page load. This means follow-up questions work naturally: "What about the second one?" refers to the previous list without the user restating context.
5
Speech Synthesis — Avatar Speaks

Once the reply text arrives, the browser calls speakTextAsync(text) on the synthesizer. Azure receives the text, synthesizes it with the configured Neural TTS voice (en-US-JennyNeural), and delivers the audio via the existing WebRTC stream while simultaneously animating Lisa's mouth movements.

Critical: Immediate Resolution
speakTextAsync() resolves its promise as soon as the speech request is queued — not when Lisa finishes speaking. This means you cannot use await speakTextAsync() to detect when speech ends. Any code placed after it runs while Lisa is still talking.
Text Preprocessing
Before passing to Azure TTS, the text is cleaned: markdown images, links, headings, bold/italic markers, and URLs are stripped. The result is truncated to 700 characters so Lisa never attempts to speak an entire travel guide aloud.
Fallback — Browser TTS
If azureSpeak() returns false (synthesizer not ready, WebRTC not connected, or exception), the app falls back to the browser's built-in SpeechSynthesisUtterance. The app prefers a male English voice so it sounds consistent. This ensures the user always gets a spoken response even if the Azure connection hasn't established yet.
6
Idle State — After Speaking

When Lisa finishes speaking, Azure fires the avatarEventReceived callback with a description containing SwitchedToIdle or TalkingEnded. This is the only reliable signal that speech has finished. The UI resets — status badge clears, waveform stops, stop button hides — but the video stream is untouched.

What triggers it
The event description is checked for three substrings: switchedtoidle, talkingended, or idle. Azure may change exact event names between SDK versions, so checking all three makes the handler resilient.
What does NOT happen
The video element is not hidden. Azure continues streaming Lisa's idle pose animation over WebRTC. Hiding the video on idle would produce a blank frame — an early bug that was fixed by removing all video display toggling from this handler.
ICE Failure — The Only True Fallback
The video is only ever hidden in one scenario: peerConn.iceConnectionState becomes failed or disconnected. This means the WebRTC connection itself dropped — likely a network interruption. At that point the static images.webp placeholder is shown, azureReady is set to false, and browser TTS takes over for any subsequent replies.
